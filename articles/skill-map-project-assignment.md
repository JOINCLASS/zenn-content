# はじめに: 「この人、空いてるから」でアサインしていませんか?

プロジェクトマネージャーの皆さん、こんな経験はありませんか?

- 「Aさん、今週空いてるからこの案件お願い」→ 実はReactは触ったことがなく、学習コストで遅延
- 新規案件のキックオフで「誰がこの技術できる?」と聞いても、正確な答えが返ってこない
- ベテランエンジニアに案件が集中し、ジュニアは成長機会を逃している
- 「うちのチーム、誰が何できるんだっけ...?」と毎回手探り

これらは全て、**チーム全体のスキル状況が可視化されていない**ことが原因です。

私自身、前職で15人規模の開発チームをマネジメントしていた際、エンジニアのスキルセットを正確に把握できておらず、プロジェクトアサインのたびに個別ヒアリングを繰り返していました。結果として、アサイン決定に平均2-3日かかり、その間に最適なタイミングを逃すこともしばしば。

本記事では、**従業員スキルマップ**という手法を使って、チームのスキルを可視化し、プロジェクトアサインを最適化する具体的な方法を解説します。明日から実践できる5ステップと、運用時の落とし穴も併せてご紹介します。

# スキルマップとは何か

## 定義

**スキルマップ**とは、組織やチームに所属するメンバーが保有するスキル・知識・経験を一覧化し、可視化したものです。

例えば、以下のような情報を体系的に整理します:

- **技術スキル**: プログラミング言語(JavaScript, Python, Go...)、フレームワーク(React, Vue, Django...)、インフラ(AWS, Docker, Kubernetes...)
- **業務スキル**: 要件定義、設計、テスト、運用保守、ドメイン知識(金融、医療、EC...)
- **ソフトスキル**: リーダーシップ、メンタリング、プレゼンテーション、交渉力

## 可視化のイメージ

スキルマップの可視化方法は様々ですが、代表的なものを2つ紹介します。

### 1. スキルマトリクス表

```
                  | React | Vue | TypeScript | AWS | Docker | 要件定義 | レビュー
----------------- | ----- | --- | ---------- | --- | ------ | ------- | --------
田中 太郎(5年目)   |   5   |  3  |     4      |  4  |   3    |    4    |    5
佐藤 花子(3年目)   |   4   |  -  |     4      |  2  |   2    |    2    |    3
鈴木 次郎(1年目)   |   2   |  -  |     2      |  1  |   1    |    1    |    1
```

※ レベル: 1=初学者、2=基礎理解、3=実務経験あり、4=高度な実装可能、5=チームリーダー・エキスパート

### 2. レーダーチャート

個人のスキルバランスを視覚的に把握できます。

```
        React(5)
           /\
          /  \
    AWS(4)    TypeScript(4)
        |      |
        |      |
  Docker(3)--要件定義(4)
```

## なぜスキルマップが必要なのか

1. **アサイン精度の向上**: 「誰が何をできるか」が明確になり、適材適所の配置が可能に
2. **育成計画の明確化**: スキルギャップが可視化され、研修やOJTの優先順位が決めやすい
3. **リスク管理**: 特定スキルの属人化(「この技術、Aさんしかできない」)を早期発見
4. **採用戦略**: 組織全体で不足しているスキルが明確になり、採用要件が具体化
5. **コミュニケーション円滑化**: 「このスキル、誰に聞けばいい?」がすぐわかる

# スキルマップの作り方(5ステップ)

それでは、実際にスキルマップを作成する手順を見ていきましょう。

## ステップ1: スキルカテゴリの定義

まず、「何のスキルを管理するか」を決めます。全てのスキルを網羅しようとすると破綻するため、**現在のビジネスに直結するスキル**に絞るのがポイントです。

### 実践例: SaaS開発企業の場合

```markdown
【技術スキル - フロントエンド】
- React, Vue, Angular
- TypeScript, JavaScript
- CSS設計(BEM, CSS Modules...)

【技術スキル - バックエンド】
- Node.js, Python, Go, Ruby
- REST API設計, GraphQL
- データベース設計(PostgreSQL, MySQL)

【技術スキル - インフラ】
- AWS(EC2, S3, RDS, Lambda...)
- Docker, Kubernetes
- CI/CD(GitHub Actions, CircleCI)

【業務スキル】
- 要件定義
- UI/UX設計
- テスト設計
- パフォーマンスチューニング

【ドメイン知識】
- 決済システム
- 認証・認可
- SaaS特有の課題(マルチテナント、課金...)

【ソフトスキル】
- テックリード経験
- メンタリング
- 採用面接
- 顧客折衝
```

**注意点:**
- 最初は**10-15カテゴリ程度**に絞る(多すぎると運用が破綻)
- チーム全体で合意形成する(マネージャーの独断で決めない)
- 3-6ヶ月ごとに見直す(技術トレンドや事業方針の変化に対応)

## ステップ2: レベル基準の設定(1-5段階)

次に、各スキルの**習熟度を測る基準**を定義します。ここが曖昧だと評価がブレるため、具体的な行動ベースで定義します。

### レベル定義の例(React)

| レベル | 定義 | 具体的な行動例 |
|-------|------|--------------|
| 1 | 初学者 | チュートリアルを完了した。公式ドキュメントを読んでいる。 |
| 2 | 基礎理解 | 簡単なコンポーネントを実装できる。propsやstateの概念を理解している。 |
| 3 | 実務経験あり | 実プロジェクトで機能開発を担当。Hooksを使いこなせる。 |
| 4 | 高度な実装可能 | パフォーマンス最適化、複雑な状態管理(Redux, Zustand)、テスト設計ができる。 |
| 5 | エキスパート | アーキテクチャ設計、ベストプラクティスの策定、他メンバーへの技術指導ができる。 |

### レベル定義のポイント

- **行動ベースで記述**: 「理解している」ではなく「〇〇ができる」
- **プロジェクト経験を基準に**: 「実際にプロダクション環境で使った」を重視
- **全スキル共通の基準も用意**: 個別定義が難しいスキルは汎用基準を使う

```markdown
【汎用レベル基準】
1. 学習中(書籍・チュートリアル)
2. 基礎知識あり(簡単なタスクをサポート付きで実行可能)
3. 実務経験あり(独力でタスク完遂可能)
4. 応用力あり(複雑な課題を解決、他者をサポート可能)
5. エキスパート(設計・意思決定、チーム全体をリード可能)
```

## ステップ3: 自己評価 + 上長評価

評価方法は以下の2段階をおすすめします。

### 3-1. 自己評価(セルフアセスメント)

まずメンバー自身に評価してもらいます。Googleフォームやスプレッドシートで実施可能です。

```markdown
【自己評価シートの例】

■ 技術スキル
- React: [1 / 2 / 3 / 4 / 5]
- TypeScript: [1 / 2 / 3 / 4 / 5]
...

■ 学びたいスキル(自由記述)
- 今後伸ばしたいスキル:
- 理由:

■ 強みと弱み(自由記述)
- 自分の強みだと思うスキル:
- 補強したい領域:
```

### 3-2. 上長評価(キャリブレーション)

自己評価は**過小評価**や**過大評価**が発生しがちです。そのため、直属の上長やテックリードが以下を確認します:

- プロジェクト実績と整合性があるか
- 他のメンバーと比較して妥当か
- 成長曲線(半年前との比較)は適切か

**1on1の場で対話**しながら最終決定するのが理想です。

```
例:
メンバーA「Reactは4だと思います」
→ 上長「確かに実装力は高いけど、まだアーキテクチャ設計の経験がないから3.5くらいかな。次のプロジェクトで設計を担当してもらえば4になれるよ」
```

## ステップ4: 可視化(レーダーチャート等)

評価データが揃ったら、可視化します。

### ツールの選択肢

1. **スプレッドシート**(Excel, Googleスプレッドシート)
   - 手軽に始められる
   - 条件付き書式でヒートマップ化も可能
   - 欠点: 更新が手動、検索性が低い

2. **BI ツール**(Tableau, Looker, Power BI)
   - 高度な分析が可能
   - インタラクティブなダッシュボード
   - 欠点: コスト、学習コスト

3. **専用ツール**(スキル管理SaaS)
   - スキルマップ専用設計
   - 更新通知、承認フロー、レポート機能
   - 欠点: 導入コスト

### 可視化の実例

#### チーム全体のスキルマトリクス

```
                  | React | TypeScript | AWS | 要件定義 | 平均
----------------- | ----- | ---------- | --- | ------- | ----
田中 太郎(TL)      |   5   |     4      |  4  |    5    | 4.5
佐藤 花子(Senior)  |   4   |     4      |  3  |    3    | 3.5
鈴木 次郎(Junior)  |   2   |     2      |  1  |    1    | 1.5
山田 美咲(Mid)     |   3   |     3      |  2  |    2    | 2.5
----------------- | ----- | ---------- | --- | ------- | ----
チーム平均         |  3.5  |    3.25    | 2.5 |   2.75  | 3.0
```

> **気づき**: AWSのスキルが全体的に低い。次の四半期で研修を実施すべき。

#### 個人のレーダーチャート(田中さん)

```
         React(5)
            /\
           /  \
     AWS(4)    TypeScript(4)
         |      |
         |      |
  Docker(3)--要件定義(5)
```

> **気づき**: Dockerが相対的に弱い。インフラ案件でキャッチアップのチャンス。

## ステップ5: プロジェクトアサインへの活用

いよいよスキルマップを実際のアサインに活用します。

### アサインの判断基準

新規プロジェクトの要求スキルと、メンバーのスキルマップを照合します。

**例: 新規プロジェクト「顧客管理ダッシュボード開発」**

```markdown
【必須スキル】
- React: Lv3以上
- TypeScript: Lv3以上
- REST API設計: Lv3以上

【推奨スキル】
- UI/UX設計: Lv2以上
- テスト設計: Lv3以上
```

**スキルマップと照合**

| メンバー | React | TypeScript | API設計 | UI/UX | テスト | 適合度 |
|---------|-------|-----------|---------|-------|-------|-------|
| 田中    |   5   |     4     |    5    |   4   |   5   | ◎ 過剰 |
| 佐藤    |   4   |     4     |    4    |   3   |   4   | ◎ 最適 |
| 山田    |   3   |     3     |    3    |   2   |   2   | ○ ギリギリ |
| 鈴木    |   2   |     2     |    1    |   1   |   1   | × 不足 |

**判断例:**
- **佐藤さんをリード**に配置(スキル最適 + 成長機会)
- **山田さんをサブ**に配置(実務経験を積ませる)
- **鈴木さんはペア**で参加(UI実装の一部を担当、佐藤さんがメンター)
- **田中さんは別の高難度案件**に配置(スキルの有効活用)

### アサイン最適化のポイント

1. **「ちょうど良い負荷」を意識**: 常に快適な案件だと成長しない。少しストレッチする案件を意図的に割り当てる
2. **属人化の解消**: 「この技術は〇〇さんしかできない」状態を避ける。レベル5の人とレベル2の人をペアにする
3. **キャリア希望を考慮**: スキルマップと併せて「学びたいスキル」も確認し、成長機会を提供

# スキルマップがプロジェクトアサインを変える具体例

実際にスキルマップを導入した架空のチームの事例を紹介します。

## 【導入前】属人化と機会損失の悪循環

**状況:**
- 10人規模の開発チーム
- プロジェクトアサインはマネージャーの感覚頼り
- 「誰が何をできるか」が不透明

**問題:**

1. **ベテランに案件集中**
   - 田中さん(5年目)にReact案件が集中
   - 「React = 田中さん」が常態化
   - 田中さんは新しいスキルを学ぶ時間がない

2. **ジュニアの成長機会不足**
   - 鈴木さん(1年目)は簡単なタスクのみ
   - 「できるかわからないから、安全な人に振ろう」という判断
   - 鈴木さんのスキルレベルは半年間ほぼ横ばい

3. **緊急時の対応困難**
   - 田中さんが病欠 → React案件が止まる
   - 代替要員がいない

4. **採用のミスマッチ**
   - 「フロントエンド経験者」を採用 → 実はチームは今バックエンドが不足していた
   - 組織全体のスキルギャップを把握していなかった

## 【導入後】データドリブンなアサイン

スキルマップを導入し、3ヶ月後の変化:

**改善1: 計画的な育成とリスク分散**

```
【React案件のアサイン方針】
- リード: 佐藤さん(Lv4) ← 田中さんから権限移譲
- サブ: 山田さん(Lv3) ← 実務経験を積む
- ペア: 鈴木さん(Lv2) ← 山田さんとペアプロ

→ 田中さんは新技術(Go言語)の案件にアサイン
→ React案件の属人化を解消しつつ、チーム全体のスキルレベルが向上
```

**改善2: 「ストレッチアサイン」の設計**

山田さん(Lv3)に対し、少し難易度の高いUI設計を任せる。

```
山田さんのスキルマップ:
- React: 3 → 目標: 4
- UI/UX設計: 2 → 目標: 3

アサイン:
- 「顧客ダッシュボードのUI設計」をリード
- 佐藤さんが週1でレビュー

3ヶ月後:
- React: 3 → 4に成長
- UI/UX設計: 2 → 3に成長
- 自信がついて、次はテックリードに挑戦したいと発言
```

**改善3: データに基づく採用戦略**

スキルマップを分析した結果、以下が判明:

```
【チーム全体の弱点】
- インフラスキル(AWS, Kubernetes)が全員Lv3以下
- 決済システムのドメイン知識を持つ人がゼロ

【採用要件の変更】
変更前: 「React経験3年以上」
変更後: 「AWSでの運用経験2年以上、決済システム経験者優遇」
```

> 組織のスキルバランスを考慮した採用が可能に

## 成果の定量化

| 指標 | 導入前 | 導入後(3ヶ月) | 改善 |
|-----|-------|-------------|-----|
| アサイン決定時間 | 平均2-3日 | 平均0.5日 | **83%削減** |
| スキル属人化率 | 5スキル | 1スキル | **80%削減** |
| ジュニアのスキルアップ速度 | +0.3/半年 | +1.2/半年 | **4倍** |
| 緊急時の代替要員確保率 | 40% | 90% | **2.25倍** |

# スキルマップ運用の落とし穴と対策

スキルマップは導入すれば終わりではなく、**継続的な運用**が鍵です。しかし、運用には典型的な失敗パターンがあります。

## 落とし穴1: 更新されずに放置される

**問題:**
- 最初は頑張って作成 → 3ヶ月後には古い情報だらけ
- 「更新するの面倒...」で誰も触らなくなる

**対策:**
- **定期更新のルール化**: 四半期ごとの1on1で必ずスキル評価を実施
- **軽量化**: 更新が15分以内で終わるシンプルな設計に
- **ツール活用**: 手動更新が不要な専用ツールを検討

```markdown
【運用カレンダー例】
- 1月, 4月, 7月, 10月: 全社スキル評価週間
- 各マネージャーは1on1でメンバーのスキル評価を更新
- 更新完了率を経営会議で報告
```

## 落とし穴2: 評価基準が曖昧で一貫性がない

**問題:**
- チームAではReact Lv3 = 「実務経験1年」
- チームBではReact Lv3 = 「実務経験3年 + 設計経験」
- 横断アサインで混乱が発生

**対策:**
- **組織共通の評価基準を文書化**: マネージャー全員が参照できる場所に
- **キャリブレーション会議**: 複数のマネージャーで評価をすり合わせ
- **実績ベースの評価**: 「〇〇プロジェクトで△△を達成」という事実を重視

## 落とし穴3: スキルマップが「監視ツール」になる

**問題:**
- 「スキルレベルが低い = 評価が下がる」という誤解
- メンバーが自己防衛で過大評価 or 評価を拒否

**対策:**
- **成長支援ツールであることを強調**: 「弱点を見つけて攻撃する」のではなく「成長機会を見つけるため」
- **レベル1-2も価値があることを示す**: 「初学者こそ伸びしろが大きい」
- **評価とは切り離す**: 人事評価とスキルマップは別物(スキルが高い = 給料が高いではない)

```markdown
【マネージャーからのメッセージ例】

「スキルマップは、みんなの成長を支援するためのツールです。
レベル1や2だからといって評価が下がることは一切ありません。
むしろ、『ここを伸ばしたい』という意志があることが重要です。

一緒に、あなたのキャリアをデザインしていきましょう。」
```

## 落とし穴4: スキルマップだけでアサインを決める

**問題:**
- 「スキルが足りないから、この案件は任せられない」と機械的に判断
- 本人の希望や、キャリアの方向性を無視

**対策:**
- **スキルマップは判断材料の1つ**: 本人の希望、チーム状況、緊急度なども総合的に考慮
- **ストレッチ機会を意図的に作る**: 「今のスキルより少し上」の案件も積極的に割り当てる
- **対話を重視**: 「このスキル、伸ばしたい?」をヒアリング

## 落とし穴5: カテゴリが多すぎて管理不能に

**問題:**
- 「あらゆるスキルを網羅しよう」→ 50カテゴリ
- 更新が追いつかず、形骸化

**対策:**
- **最小限から始める**: 10-15カテゴリで十分
- **優先度を決める**: 「今の事業に直結するスキル」だけに絞る
- **定期的に棚卸し**: 使われていないカテゴリは削除

# まとめ: ツールを使ってスキルマップを次のレベルへ

本記事では、従業員スキルマップを活用したプロジェクトアサインの最適化について解説しました。

## 重要ポイントの振り返り

1. **スキルマップは「可視化」が目的ではなく「アクション」が目的**
   - アサイン最適化、育成計画、リスク管理に活用してこそ価値がある

2. **5ステップで実践可能**
   - カテゴリ定義 → レベル基準 → 評価 → 可視化 → アサイン活用

3. **運用が全て**
   - 作って終わりではなく、定期更新とチーム全体の合意形成が鍵

4. **成長支援ツールとして位置づける**
   - 監視や評価のためではなく、メンバーのキャリア形成を支援するために使う

## スプレッドシートから始めよう

「明日から始められる」と言いましたが、実際にはスプレッドシートで十分スタートできます。

```markdown
【最小構成のスキルマップ】
1. Googleスプレッドシートを作成
2. 縦軸: メンバー名、横軸: スキルカテゴリ(10個)
3. 各セルに1-5の数値を入力
4. 条件付き書式でヒートマップ化(5=緑、1=赤)
5. 月1回の1on1で更新

→ これだけでも十分に効果があります。
```

## さらに効率化したい方へ

ただし、チームが10人を超えてくると、スプレッドシート運用には限界があります:

- 更新の手間が増える
- 履歴管理ができない
- プロジェクトとの紐付けが難しい
- レポート作成が手作業

そんな時は、**専用のスキル管理ツール**を検討してみてください。

スキル評価のワークフロー自動化、プロジェクトとスキルの自動マッチング、スキルギャップのレポート自動生成、成長履歴のトラッキングなど、運用負荷を大幅に削減できます。

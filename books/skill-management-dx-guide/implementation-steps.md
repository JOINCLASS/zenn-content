## 12週間で全社展開するロードマップ

前チャプターまでで、スキルマップの設計方法、活用方法、ツールの選び方を解説してきました。しかし、「やり方は分かったけど、実際にどう進めればいいのか」が最大のハードルです。

このチャプターでは、スキル管理DXを**12週間（約3か月）で全社展開するための具体的なステップ**を解説します。一気に全社導入するのではなく、小さく始めて成功体験を積み、段階的に広げていくアプローチです。

## 全体スケジュール

```
Phase 1（1-2週目）: 現状の棚卸しとスキルカテゴリ設計
Phase 2（3-4週目）: ツール選定と初期データ投入
Phase 3（5-8週目）: パイロット運用（1チーム）
Phase 4（9-12週目）: 全社展開と運用定着
```

## Phase 1: 現状の棚卸しとスキルカテゴリ設計（1-2週目）

### やること

**第1週: 現状の棚卸し**

まず、今のスキル管理がどういう状態にあるかを正直に把握します。

```
【棚卸しチェックリスト】
- [ ] スキル情報は何で管理しているか?（エクセル? 紙? 頭の中?）
- [ ] 最後に更新されたのはいつか?
- [ ] 何人分のデータがあるか?
- [ ] データの正確性は何%くらいか?（体感で構わない）
- [ ] 今、誰がスキル管理を担当しているか?
- [ ] スキルデータは実際に何に使われているか?（使われていない場合はそれも記録）
```

**第2週: スキルカテゴリとレベル定義の設計**

第3チャプターで解説した設計フレームワークに基づき、自社用のスキルカテゴリとレベル定義を作成します。

```
【設計ワークショップの進め方（所要時間: 2時間）】

参加者: 人事担当、マネージャー2-3名、シニアエンジニア1-2名

1. ブレスト（30分）
   - 「自社で重要なスキルは何か?」を付箋に書き出す
   - 1人10枚以上を目標に

2. グルーピング（30分）
   - テクニカル / ソフト / ドメインの3軸で分類
   - 重複を統合、抽象度を揃える

3. 優先度決定（30分）
   - 「今の事業に直結するか」で10-15個に絞る
   - 全員の合意を取る（多数決ではなく対話で）

4. レベル定義（30分）
   - 汎用基準を確認
   - 上位3-5スキルは個別のレベル定義を作成
```

### Phase 1の完了基準

- [ ] スキルカテゴリ一覧（10〜15個）が確定している
- [ ] 汎用レベル基準（1-5段階）が文書化されている
- [ ] 主要スキル3〜5個の個別レベル定義が完成している
- [ ] 推進担当者（プロジェクトオーナー）が決まっている

## Phase 2: ツール選定と初期データ投入（3-4週目）

### やること

**第3週: ツール選定**

前チャプターの判断フローに基づき、自社に合ったツールを選定します。

```
【ツール比較テンプレート】

評価項目（各5点満点）:
                    | ツールA | ツールB | ツールC |
UIのシンプルさ        |        |        |        |
カスタマイズ性        |        |        |        |
コスト（年間）        |        |        |        |
導入サポート          |        |        |        |
セキュリティ          |        |        |        |
既存ツールとの連携     |        |        |        |
合計                 |        |        |        |
```

ポイントは、**2週間以内に決断すること**です。完璧なツールは存在しません。「70点で十分」と割り切り、まず使い始めることが重要です。

**第4週: 初期データ投入**

選定したツール（またはスプレッドシート）に、Phase 1で設計したカテゴリとレベル定義を設定し、パイロットチームのメンバー情報を登録します。

```
【初期データ投入の手順】
1. スキルカテゴリの登録（15分）
2. レベル定義の登録（15分）
3. パイロットチームのメンバーアカウント作成（30分）
4. 管理者権限の設定（15分）
5. テストデータでの動作確認（30分）
```

### Phase 2の完了基準

- [ ] ツール（またはスプレッドシート）が決定している
- [ ] スキルカテゴリとレベル定義がツールに登録されている
- [ ] パイロットチームのメンバーがアカウントを持っている
- [ ] 管理者が基本操作を理解している

## Phase 3: パイロット運用（5-8週目）

### なぜパイロットが必要なのか

いきなり全社展開すると、以下のリスクがあります。

- 設計の不備が全社に波及する
- 現場の反発を受けて頓挫する
- 問題が起きたときの影響範囲が大きすぎる

**まず1チーム（5-10名）で試し、成功事例を作ってから広げる**のが鉄則です。

### パイロットチームの選び方

```
【理想的なパイロットチームの条件】
- 5-10名程度の規模
- マネージャーが協力的（これが最重要）
- メンバーのITリテラシーが比較的高い
- 直近でプロジェクトアサインの機会がある
```

### 4週間の運用スケジュール

**第5週: キックオフと初回入力**

```
キックオフミーティング（30分）:
1. スキル管理DXの目的を説明（5分）
2. ツールの使い方をデモ（10分）
3. 自己評価を入力してもらう（10分）
4. 質疑応答（5分）

重要なメッセージ:
「これは監視ではなく、皆さんの成長を支援するためのツールです。
完璧な入力は求めていません。まずは"だいたい"でOKです。」
```

**第6週: マネージャーによるキャリブレーション**

マネージャーが各メンバーの自己評価を確認し、1on1で調整します。

```
1on1での確認ポイント（1人15分）:
- 自己評価と上長評価の差異はないか?
- 特に伸ばしたいスキルはあるか?
- ツールの使い勝手はどうか?（フィードバック収集）
```

**第7週: アサインへの活用テスト**

実際のプロジェクトアサインに、スキルマップを使ってみます。

```
テスト項目:
- プロジェクト要件とスキルマップの照合ができるか?
- アサイン判断がスムーズになったか?
- 不足している情報はないか?
```

**第8週: 振り返りと改善**

パイロット運用の振り返りを行い、全社展開前に改善します。

```
【振り返りミーティング（1時間）】

参加者: パイロットチーム全員

1. 良かった点（15分）
   - ツールの使い勝手で良い部分
   - 実際に役立った場面

2. 改善点（15分）
   - 使いにくかった部分
   - カテゴリやレベル定義の修正点
   - 運用ルールの見直し

3. 定量フィードバック（15分）
   - 更新率: 何%のメンバーが入力を完了したか
   - 所要時間: 1回の更新に何分かかったか
   - 満足度: 5段階で評価

4. 改善アクションの決定（15分）
   - 全社展開前に修正すべき点をリスト化
```

### Phase 3の完了基準

- [ ] パイロットチームの更新率が80%以上
- [ ] 1回の更新が15分以内に収まっている
- [ ] 実際のアサイン判断に活用できることを確認した
- [ ] フィードバックに基づく改善が完了している

## Phase 4: 全社展開と運用定着（9-12週目）

### やること

**第9-10週: 全社展開**

パイロットの成功事例を元に、全社に展開します。

```
【全社展開のステップ】

1. 経営層への報告（第9週前半）
   - パイロットの成果を数字で示す
   - 全社展開の計画と予算を承認してもらう

2. マネージャー向け説明会（第9週後半）
   - パイロットの成功事例を共有
   - 各チームでの展開手順を説明
   - マネージャー自身がまず使えるようになる

3. 全社キックオフ（第10週）
   - 全従業員にアカウントを配布
   - 初回入力の締切を設定（1週間以内）
   - ヘルプデスク窓口を用意
```

**第11-12週: 運用定着**

展開直後の2週間は「定着の正念場」です。以下に注力します。

```
【定着のためのアクション】

毎日:
- 更新率をモニタリング
- 未入力者へのリマインド（Slackなど）

毎週:
- FAQ・困りごとの集約と対応
- マネージャーへの進捗報告

第12週末:
- 全社の更新率を集計
- 初回の全社スキルレポートを作成
- 今後の運用カレンダーを全社に共有
```

### Phase 4の完了基準

- [ ] 全従業員のアカウントが作成されている
- [ ] 全社の初回入力率が70%以上
- [ ] 四半期ごとの運用カレンダーが設定されている
- [ ] ヘルプデスクの対応フローが確立している

## 社内説得のためのプレゼン資料の作り方

スキル管理DXを進めるには、経営層やマネージャー層の理解と協力が不可欠です。社内プレゼンで押さえるべきポイントを整理します。

### スライド構成テンプレート（5枚）

**スライド1: 現状の課題**

```
タイトル: 「スキル管理、今こうなっていませんか?」

- エクセルの最終更新日: ○月○日（X か月前）
- プロジェクトアサインにかかる時間: 平均○日
- 「誰がこの技術できる?」に即答できるか: No
- スキルの属人化: 「○○さんしかできない」が○件
```

**スライド2: 放置するとどうなるか**

```
タイトル: 「このまま放置すると...」

- アサインの失敗 → プロジェクト遅延 → 売上への影響
- 属人化 → キーパーソンの退職リスク → 事業継続リスク
- 成長機会の偏り → エンジニアの不満 → 離職率の上昇
- 採用のミスマッチ → 無駄な採用コスト
```

**スライド3: 解決策**

```
タイトル: 「スキル管理DXで解決する」

- スキルマップの設計と運用を仕組み化
- 12週間で段階的に導入（いきなり全社ではない）
- まず1チームで試し、成功を確認してから展開
```

**スライド4: コストと効果**

```
タイトル: 「投資対効果」

コスト:
- ツール費用: 月額○○円
- 導入工数: 約○○時間（12週間で分散）

期待効果:
- アサイン決定時間: 2-3日 → 半日（83%削減）
- 属人化リスク: 大幅に軽減
- 育成計画: データに基づく計画的な育成が可能に
```

**スライド5: 次のアクション**

```
タイトル: 「来週からやること」

1. パイロットチームの選定（今週中）
2. スキルカテゴリの設計ワークショップ（来週）
3. ツールの無料トライアル開始（再来週）
```

### 説得のコツ

- **数字で語る**: 「なんとなく不便」ではなく「アサインに2-3日かかっている」
- **リスクで語る**: 「もし○○さんが退職したら?」は経営層に響く
- **小さく始める提案**: 「まず1チームで2か月間試させてください」は承認を得やすい
- **他社事例を引用**: 「同規模の企業ではアサイン時間が83%短縮した例がある」

## 12週間のまとめ

| Phase | 期間 | 主なアクション | 完了基準 |
|-------|------|-------------|---------|
| 1 | 1-2週目 | 棚卸し、カテゴリ設計 | カテゴリとレベル定義が確定 |
| 2 | 3-4週目 | ツール選定、データ投入 | ツールが稼働可能な状態 |
| 3 | 5-8週目 | パイロット運用 | 更新率80%、アサイン活用を確認 |
| 4 | 9-12週目 | 全社展開、定着 | 全社更新率70%、運用カレンダー設定 |

大切なのは、**完璧を目指さず、まず動き出すこと**です。スキルマップの精度は、運用を重ねるうちに自然と上がっていきます。

次のチャプターでは、導入後の効果をどう測定し、継続的に改善していくかを解説します。

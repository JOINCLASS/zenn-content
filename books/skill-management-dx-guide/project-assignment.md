## 「この人、空いてるから」でアサインしていませんか?

プロジェクトマネージャーの皆さん、こんな経験はありませんか?

- 「Aさん、今週空いてるからこの案件お願い」 → 実はReactは触ったことがなく、学習コストで遅延
- 新規案件のキックオフで「誰がこの技術できる?」と聞いても、正確な答えが返ってこない
- ベテランエンジニアに案件が集中し、ジュニアは成長機会を逃している
- 「うちのチーム、誰が何できるんだっけ...?」と毎回手探り

これらは全て、**チーム全体のスキル状況が可視化されていない**ことが原因です。

前チャプターで設計したスキルマップを使えば、こうした問題を構造的に解決できます。本チャプターでは、スキルマップをプロジェクトアサインに活用する具体的な方法と、運用時の落とし穴について解説します。

## アサイン判断の5ステップ

スキルマップを使ったアサインは、以下の5ステップで進めます。

### ステップ1: プロジェクトの必須スキルを定義する

新規プロジェクトが立ち上がったら、まず「このプロジェクトに必要なスキルとレベル」を明確にします。

**例: 顧客管理ダッシュボード開発プロジェクト**

```
【必須スキル】
- フロントエンド開発（React）: レベル3以上
- TypeScript: レベル3以上
- REST API設計: レベル3以上

【推奨スキル】
- UI/UX設計: レベル2以上
- テスト設計: レベル3以上
```

ポイントは、**必須と推奨を分ける**ことです。全てを必須にすると該当者が見つかりません。

### ステップ2: スキルマップと照合する

定義した要件を、チームのスキルマップと照合します。

| メンバー | React | TypeScript | API設計 | UI/UX | テスト | 適合度 |
|---------|-------|-----------|---------|-------|-------|-------|
| 田中(TL) |   5   |     4     |    5    |   4   |   5   | 過剰 |
| 佐藤(Sr) |   4   |     4     |    4    |   3   |   4   | 最適 |
| 山田(Mid) |   3   |     3     |    3    |   2   |   2   | ギリギリ |
| 鈴木(Jr) |   2   |     2     |    1    |   1   |   1   | 不足 |

### ステップ3: 「成長機会」を加味する

スキルの適合度だけで判断すると、常にベテランに案件が集中します。**意図的に「少し背伸びする」アサイン**を設計することが重要です。

**判断例:**

- **佐藤さんをリード**に配置 → スキルが最適で、リーダー経験を積む成長機会
- **山田さんをサブ**に配置 → 現スキルで対応可能だが、UI/UXは伸びしろ
- **鈴木さんはペア**で参加 → UI実装の一部を山田さんとペアプロで担当
- **田中さんは別の高難度案件**に配置 → スキルの有効活用、属人化の解消

### ステップ4: 本人の希望を確認する

スキルマップはあくまで「判断材料の一つ」です。本人のキャリア希望も必ず確認します。

```
確認すべきこと:
- 「この案件に興味があるか?」
- 「今、伸ばしたいスキルは何か?」
- 「負荷感はどうか?（他の案件との兼ね合い）」
```

本人が望まないアサインはモチベーション低下につながります。数字だけで機械的に決めないことが大切です。

### ステップ5: フォローアップ体制を設計する

特にストレッチアサイン（現スキルより少し高い案件への配置）の場合は、フォロー体制を明確にしておきます。

```
フォローアップ設計の例:
- 週1回、シニアメンバーとの技術レビュー
- 困ったらすぐ相談できるメンターの指名
- 2週間ごとの振り返り（スキルの伸び具合を確認）
```

## 導入前後の変化 --- 具体例で見る

実際にスキルマップベースのアサインを導入するとどのような変化が起きるのか、10人規模の開発チームの事例で見てみましょう。

### 導入前: 属人化と機会損失の悪循環

**起きていた問題:**

**1. ベテランに案件集中**
田中さん（5年目）にReact案件が集中。「React = 田中さん」が常態化し、田中さんは新しいスキルを学ぶ時間がない。

**2. ジュニアの成長機会不足**
鈴木さん（1年目）は簡単なタスクのみ。「できるかわからないから安全な人に」という判断で、スキルレベルは半年間ほぼ横ばい。

**3. 緊急時の対応困難**
田中さんが病欠するとReact案件が止まる。代替要員がいない。

**4. 採用のミスマッチ**
「フロントエンド経験者」を採用したが、実はチームに不足していたのはバックエンドだった。

### 導入後: データドリブンなアサイン（3か月後）

**改善1: 計画的な育成とリスク分散**

```
【React案件のアサイン方針を変更】
- リード: 佐藤さん(レベル4) ← 田中さんから権限移譲
- サブ: 山田さん(レベル3) ← 実務経験を積む
- ペア: 鈴木さん(レベル2) ← 山田さんとペアプロ

→ 田中さんは新技術(Go言語)の案件にアサイン
→ React案件の属人化を解消しつつ、チーム全体のスキルレベルが向上
```

**改善2: 「ストレッチアサイン」の実践**

山田さん（レベル3）に、少し難易度の高いUI設計を任せた例です。

```
山田さんのスキル目標:
- React: 3 → 目標4
- UI/UX設計: 2 → 目標3

アサイン内容:
- 「顧客ダッシュボードのUI設計」をリード
- 佐藤さんが週1でレビュー

3か月後の結果:
- React: 3 → 4に成長
- UI/UX設計: 2 → 3に成長
- 自信がつき、次はテックリードに挑戦したいと発言
```

**改善3: データに基づく採用戦略**

スキルマップのギャップ分析から、組織の弱点が判明しました。

```
【チーム全体の弱点】
- インフラスキル（AWS, Kubernetes）が全員レベル3以下
- 決済システムのドメイン知識を持つ人がゼロ

【採用要件の変更】
変更前: 「React経験3年以上」
変更後: 「AWSでの運用経験2年以上、決済システム経験者優遇」
```

### 成果の定量化

| 指標 | 導入前 | 導入後（3か月） | 改善率 |
|-----|-------|-------------|-------|
| アサイン決定時間 | 平均2〜3日 | 平均半日 | 83%削減 |
| スキル属人化数 | 5スキル | 1スキル | 80%削減 |
| ジュニアのスキルアップ速度 | +0.3/半年 | +1.2/半年 | 4倍 |
| 緊急時の代替要員確保率 | 40% | 90% | 2.25倍 |

## アサイン運用の落とし穴と対策

### 落とし穴1: スキルマップだけで機械的に決める

**問題:** 「スキルが足りないから任せられない」と数値だけで判断してしまう。

**対策:** スキルマップは判断材料の一つ。本人の希望、チーム状況、プロジェクトの緊急度も総合的に考慮します。

### 落とし穴2: チーム間で評価基準がバラバラ

**問題:** チームAとチームBで同じレベル3の意味が異なり、横断アサインで混乱が発生。

**対策:** 組織共通の評価基準を文書化し、複数マネージャーでキャリブレーション（すり合わせ）会議を実施します。

### 落とし穴3: 「監視ツール」だと誤解される

**問題:** メンバーが「スキルレベルが低い = 評価が下がる」と感じ、過大申告や評価拒否が起きる。

**対策:** スキルマップは**成長支援ツール**であることを繰り返し伝えます。

```
【マネージャーからのメッセージ例】

「スキルマップは、みんなの成長を支援するためのツールです。
レベル1や2だからといって評価が下がることは一切ありません。
むしろ、『ここを伸ばしたい』という意志があることが重要です。
一緒に、あなたのキャリアをデザインしていきましょう。」
```

### 落とし穴4: 更新が止まり情報が陳腐化する

**問題:** 導入直後は熱心に更新していたが、3か月後には放置。

**対策:** 四半期ごとの1on1と連動させ、更新率をKPIとしてトラッキングします。

## アサイン最適化の3原則

本チャプターの内容を3つの原則にまとめます。

1. **「ちょうど良い負荷」を意識する** --- 常に快適な案件だと成長しない。少しストレッチする案件を意図的に割り当てる
2. **属人化の解消を常に意識する** --- レベル5の人とレベル2の人をペアにし、知識の移転を促進する
3. **対話を重視する** --- 数字だけで決めず、「このスキル、伸ばしたい?」を必ずヒアリングする

スキルマップの活用イメージがつかめたところで、次のチャプターでは「どのツールを使うか」というツール選定のガイドに入ります。

## スキルマップとは何か

前チャプターで、エクセル管理の限界を見てきました。では、エクセルの代わりに何をすればいいのでしょうか。

答えは、**正しく設計されたスキルマップを作ること**です。

スキルマップとは、組織やチームに所属するメンバーが保有するスキル・知識・経験を一覧化し、可視化したものです。ただし、「一覧にするだけ」では前チャプターで見たエクセルの二の舞です。重要なのは、**運用が継続する設計**にすることです。

このチャプターでは、スキルマップを「作って終わり」にしないための設計フレームワークを、4つのステップに分けて解説します。

## ステップ1: スキルカテゴリの体系化

### 3つの大分類から始める

スキルカテゴリは、以下の3つの大分類から設計するのが定石です。

**1. テクニカルスキル（技術力）**

業務で直接使う技術的な知識・能力です。IT企業であれば、プログラミング言語、フレームワーク、インフラ技術、開発ツールなどが該当します。

```
【テクニカルスキルの例】
- フロントエンド: React, Vue, Angular, TypeScript, CSS設計
- バックエンド: Node.js, Python, Go, Ruby, REST API設計, GraphQL
- インフラ: AWS, GCP, Docker, Kubernetes, CI/CD
- データ: SQL, データベース設計, データ分析
- モバイル: Swift, Kotlin, React Native, Flutter
```

**2. ソフトスキル（対人・組織力）**

技術以外の、仕事を円滑に進めるための能力です。マネジメント層ほど重要度が増します。

```
【ソフトスキルの例】
- リーダーシップ / テックリード経験
- メンタリング / 後輩育成
- プレゼンテーション / 社外発表
- プロジェクトマネジメント
- 顧客折衝 / 要件ヒアリング
- 採用面接
```

**3. ドメインスキル（業界・業務知識）**

特定の業界や業務領域に関する知識です。プロジェクトアサインの際に、技術力だけでなくドメイン知識の有無が重要になる場面は多くあります。

```
【ドメインスキルの例】
- 決済・金融システム
- 医療・ヘルスケア
- ECサイト運営
- SaaS（マルチテナント、課金設計、SLA管理）
- 認証・認可（OAuth, SAML, OpenID Connect）
```

### カテゴリ数の目安

最も重要な設計判断は、**カテゴリ数をいくつにするか**です。

| カテゴリ数 | 評価 | 理由 |
|-----------|------|------|
| 5〜10個 | スタートに最適 | 管理負荷が低く、現場が受け入れやすい |
| 10〜20個 | 標準的 | バランスが取れた運用が可能 |
| 20〜30個 | やや多い | 更新の手間が増え、形骸化リスクあり |
| 30個以上 | 非推奨 | 運用が破綻する可能性が高い |

**まずは10〜15カテゴリで始めることを強く推奨します。** 足りないと感じたら後から追加できますが、多すぎるカテゴリを後から減らすのは混乱を招きます。

### カテゴリ設計テンプレート（SaaS開発企業・30名規模の例）

```
■ テクニカルスキル（8項目）
  1. フロントエンド開発（React/Vue等）
  2. バックエンド開発（Node.js/Python等）
  3. データベース設計・運用
  4. クラウドインフラ（AWS/GCP等）
  5. CI/CD・DevOps
  6. セキュリティ
  7. テスト設計・自動化
  8. パフォーマンスチューニング

■ ソフトスキル（4項目）
  9.  テックリード・設計判断
  10. メンタリング・育成
  11. プロジェクト管理
  12. 顧客折衝・要件定義

■ ドメインスキル（3項目）
  13. 自社プロダクトのドメイン知識
  14. 決済・課金システム
  15. SaaSアーキテクチャ
```

### カテゴリ選定の3つの問い

カテゴリを選ぶ際は、以下の問いで優先度を判断してください。

1. **今の事業に直結するか?** --- 「いつか使うかも」ではなく、今のプロジェクトで実際に必要なスキル
2. **アサイン判断に影響するか?** --- このスキルの有無で、プロジェクトへの配置が変わるか
3. **複数人が該当するか?** --- 1人しか持っていないニッチなスキルはカテゴリに含めず、備考欄で対応

## ステップ2: レベル定義（1〜5段階の具体的な基準）

### なぜレベル定義が重要なのか

エクセル管理でよく見る「◎、○、△」の問題点は、**基準が曖昧でブレる**ことです。Aさんの「◎」とBさんの「◎」が同じレベルとは限りません。

この問題を解消するために、**行動ベースのレベル定義**を設計します。

### 汎用レベル基準テンプレート

全スキルに適用できる汎用基準をまず用意しましょう。

| レベル | 名称 | 定義 | 行動の目安 |
|-------|------|------|-----------|
| 1 | 初学者 | 学習を始めた段階 | 書籍やチュートリアルで学習中。実務経験なし |
| 2 | 基礎理解 | 基本的な概念を理解 | 簡単なタスクをサポート付きで完遂できる |
| 3 | 実務遂行 | 実務で独力で遂行できる | 標準的なタスクを自分の判断で完遂できる |
| 4 | 応用・指導 | 高度な課題を解決し、他者を指導できる | 複雑な問題を解決し、チームメンバーを技術的にサポートできる |
| 5 | エキスパート | 設計・意思決定をリードできる | アーキテクチャ設計やベストプラクティスの策定を主導。社内外で第一人者 |

### スキル別の具体的なレベル定義例

汎用基準に加えて、主要なスキルには個別の具体例を用意すると、評価の一貫性が高まります。

**例: React のレベル定義**

| レベル | 具体的にできること |
|-------|-----------------|
| 1 | 公式チュートリアルを完了。JSXの基本を理解している |
| 2 | 簡単なコンポーネントを実装できる。propsとstateの概念を理解 |
| 3 | Hooksを使いこなし、実プロジェクトで機能開発を担当した実績がある |
| 4 | パフォーマンス最適化、複雑な状態管理、テスト設計ができる |
| 5 | アーキテクチャ設計、ライブラリ選定、チーム全体の技術指導ができる |

**例: プロジェクトマネジメント のレベル定義**

| レベル | 具体的にできること |
|-------|-----------------|
| 1 | PMの補佐として議事録作成やタスク管理を担当した |
| 2 | 5名以下の小規模プロジェクトでリーダーを経験 |
| 3 | 10名規模のプロジェクトを独力でマネジメントできる |
| 4 | 複数プロジェクトの並行管理、リスク管理、ステークホルダー調整ができる |
| 5 | 組織横断のプログラムマネジメント、PM育成を主導できる |

### レベル定義作成の3つのルール

1. **「理解している」ではなく「できる」で書く**: 「Reactを理解している」は曖昧だが、「Hooksを使ってフォームのバリデーションを実装できる」は明確
2. **プロジェクト実績に基づく**: 「3年間の経験」よりも「実プロジェクトで○○を達成」を重視
3. **レベル3を基準に設計する**: 「一人前」のラインをレベル3に設定し、上下を定義するのが分かりやすい

## ステップ3: 更新頻度と運用ルール

スキルマップは、設計よりも**運用の継続**が難しい仕組みです。以下のルールをあらかじめ決めておくことで、形骸化を防ぎます。

### 更新タイミングの設計

| タイミング | 内容 | 担当 |
|-----------|------|------|
| 四半期ごと（定期） | 全社スキル評価週間。全員が自己評価を更新 | 人事+各マネージャー |
| 1on1面談時 | マネージャーがメンバーのスキル評価を確認・調整 | マネージャー |
| プロジェクト完了時 | プロジェクトで得たスキルを反映 | メンバー本人 |
| 入社・異動時 | 新メンバーの初期スキル登録、異動者の情報更新 | 人事+マネージャー |
| 資格取得・研修修了時 | 新たに取得した資格やスキルを追加 | メンバー本人 |

### 運用カレンダーの例（四半期サイクル）

```
第1週: 人事から「スキル更新依頼」を全社に送付
第2週: メンバーがセルフ申告を完了（締切: 金曜日）
第3週: マネージャーが1on1で確認・キャリブレーション
第4週: 人事がデータを集計し、経営レポートを作成
```

### 更新が続くための3つの工夫

**1. 15分以内に終わる設計にする**

更新に30分以上かかるようでは、現場は協力してくれません。カテゴリ数を絞り、入力方法をシンプルにして、1回の更新が15分以内で完了するようにしましょう。

**2. 現場にメリットを見せる**

「更新しても自分には何も返ってこない」状態ではモチベーションが続きません。以下のような「見返り」を設計します。

- 自分のスキルレーダーチャートが見られる（成長の実感）
- 希望するスキル領域のプロジェクトにアサインされやすくなる
- キャリアの方向性について1on1で具体的に話せるようになる

**3. 完璧を求めない**

全員が全カテゴリを正確に評価する必要はありません。「ざっくり合っている」で十分です。精度は運用を重ねるうちに上がっていきます。最初から100%の正確性を求めると、誰もやらなくなります。

## ステップ4: 可視化の方法

集めたスキルデータは、見やすい形で可視化することで初めて価値を発揮します。代表的な可視化手法を3つ紹介します。

### 1. レーダーチャート（個人のスキルバランス）

個人のスキルを多角的に表示し、強みと弱みを一目で把握できます。

```
        React(5)
           /\
          /  \
    AWS(4)    TypeScript(4)
        |      |
        |      |
  Docker(3)--要件定義(4)
```

**活用シーン**: 1on1面談での成長確認、個人のキャリアプラン設計、研修の優先度判断

### 2. ヒートマップ（チーム全体のスキル分布）

チーム全体を俯瞰し、どのスキルが充実しているか、どこにギャップがあるかを把握します。

```
                  | React | TypeScript | AWS | 要件定義 | 平均
----------------- | ----- | ---------- | --- | ------- | ----
田中 太郎(TL)      |   5   |     4      |  4  |    5    | 4.5
佐藤 花子(Senior)  |   4   |     4      |  3  |    3    | 3.5
鈴木 次郎(Junior)  |   2   |     2      |  1  |    1    | 1.5
山田 美咲(Mid)     |   3   |     3      |  2  |    2    | 2.5
----------------- | ----- | ---------- | --- | ------- | ----
チーム平均         |  3.5  |    3.25    | 2.5 |   2.75  | 3.0
```

条件付き書式でヒートマップ化すると（レベル5=濃い緑、レベル1=薄い赤）、チーム全体の強み・弱みが一目で分かります。

**活用シーン**: プロジェクトアサインの判断材料、組織のスキルギャップ発見、採用要件の策定

### 3. ギャップチャート（理想と現実の差分）

プロジェクトやチームが求めるスキルレベルと、現状のレベルの差分を可視化します。

```
スキル        | 必要レベル | チーム最高 | ギャップ | 判定
React        |    4      |    5      |   +1    | OK（余裕あり）
TypeScript   |    4      |    4      |    0    | OK（ギリギリ）
Kubernetes   |    3      |    1      |   -2    | NG（要対策）
テスト設計    |    3      |    2      |   -1    | 注意（育成必要）
```

**活用シーン**: 新規プロジェクトの実行可能性判断、育成計画の優先度決定、外部リソースの必要性判断

## スキルマップ設計チェックリスト

最後に、スキルマップの設計が完了した時点で確認すべきチェックリストです。

- [ ] スキルカテゴリは10〜15個に収まっているか
- [ ] 3つの大分類（テクニカル/ソフト/ドメイン）をカバーしているか
- [ ] レベル定義は行動ベースで記述されているか
- [ ] レベル3（一人前）の基準が明確か
- [ ] 更新頻度と担当者が決まっているか
- [ ] 1回の更新が15分以内で完了する設計か
- [ ] 現場にとってのメリットが明確か
- [ ] 可視化の方法が決まっているか

すべてにチェックが入ったら、設計は完了です。次のチャプターでは、この設計を実際のプロジェクトアサインにどう活用するかを解説します。

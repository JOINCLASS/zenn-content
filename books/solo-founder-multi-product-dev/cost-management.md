## 1人法人のコスト意識

1人法人でプロダクトを運用する場合、インフラコストはそのまま利益に直結します。大企業なら「月額数万円のSaaSを導入しよう」と気軽に判断できますが、1人法人では月額1,000円の差が年間12,000円、5プロダクトなら年間60,000円の差になります。

だからといって、すべてを自前で構築してコストをゼロにするのは本末転倒です。自分の時間単価を考えれば、月額数千円のBaaSを使う方が、自前でサーバーを管理するよりはるかに安い。重要なのは「払うべきコストと避けられるコストを見極める」ことです。

## 無料枠の活用戦略

### Vercel — Hobby Plan

Vercelの無料プラン（Hobby Plan）は、個人プロジェクトに対して以下を提供します。

- 1アカウントで複数プロジェクト
- 100 GB/月のBandwidth
- Serverless Function 実行制限あり
- 自動SSL
- プレビューデプロイ

FocalizeとShareTokuの2プロダクトを無料枠内で運用するために意識しているのは、**静的アセットの最適化**です。画像はWebP形式に変換し、next/imageで自動最適化。JavaScriptバンドルはdynamic importで分割し、初期ロードサイズを最小化。これによりBandwidthの消費を抑えています。

MochiQ LPとSkill Tracker LPをVercelに載せなかったのも、無料枠の温存が理由です。

### Supabase — Free Plan

Supabaseの無料プランは、プロジェクトごとに以下を提供します。

- 500 MBのデータベースストレージ
- 2 GBのBandwidth
- 50,000の月間アクティブユーザー
- 1 GBのファイルストレージ

FocalizeとShareTokuで別々のSupabaseプロジェクトを使っているため、無料枠は実質2倍です。現在の規模ではどちらも500MBの制限にはほど遠く、有料プランへのアップグレードは当面不要です。

ただし、Supabaseの無料プランには「7日間アクティビティがないとプロジェクトが一時停止する」という制限があります。これを避けるために、GitHub Actionsで定期的にヘルスチェックを実行するワークフローを設定しています。

### Firebase — Spark Plan

Firebaseの無料プラン（Spark Plan）は以下を提供します。

- Firestore: 1 GiB ストレージ、50,000 読み取り/日、20,000 書き込み/日
- Authentication: 無制限の認証
- Cloud Messaging: 無制限のプッシュ通知
- Analytics: 無制限

元気ボタンとMochiQの現在のユーザー規模では、Firestoreの読み書き制限に余裕があります。特にCloud Messagingが無制限なのは大きい。元気ボタンのコア機能であるプッシュ通知を無料で送信できます。

### GitHub Actions — 無料枠

GitHub Actionsの無料枠は、パブリックリポジトリでは無制限、プライベートリポジトリでは月間2,000分です。5プロダクトのCIパイプラインを2,000分に収めるために、以下の工夫をしています。

- **キャッシュの活用**: `actions/cache`で`node_modules`や`.next/cache`をキャッシュし、installとビルドの時間を短縮
- **条件付き実行**: パスフィルターを設定し、関連するファイルが変更された場合のみジョブを実行
- **並列実行の最適化**: 独立したジョブは並列に実行し、トータルの実行時間を短縮

```yaml
# パスフィルターの例
on:
  push:
    paths:
      - 'src/**'
      - 'package.json'
      - 'package-lock.json'
    paths-ignore:
      - '**.md'
      - 'docs/**'
```

### Gemini API — 無料枠

Gemini APIは無料枠が提供されており、レート制限はあるものの、開発・テスト段階では十分な量のリクエストを無料で実行できます。FocalizeのAIメール解析とMochiQのクイズ生成の両方で、無料枠の範囲内で開発を進めています。

本番環境でユーザー数が増えた場合は従量課金に移行しますが、Gemini APIの単価はOpenAI GPTと比較して安価であり、コスト面でも有利です。

## コスト最適化のTips

### 1. 静的サイトはVercelに載せない

SSRが不要なサイトは、お名前.comのような安価な共用サーバーにFTPデプロイする方がコスト効率が良い。Vercelの無料枠はSSRが必要なプロダクトのために温存します。

### 2. BaaSのプロジェクトはプロダクトごとに分ける

SupabaseもFirebaseも、無料枠はプロジェクト単位で適用されます。1つのプロジェクトに全プロダクトを詰め込むと無料枠を共有してしまうため、プロダクトごとに分けます。

### 3. 画像とアセットを最適化する

- Next.jsのnext/imageで自動WebP変換
- 不要な大きさの画像をアップロードしない
- フォントはGoogle Fontsを`next/font`で最適化ロード

### 4. CIの実行時間を短縮する

GitHub Actionsの無料枠は時間ベースです。以下で節約します。

- 依存関係のキャッシュ
- パスフィルターによる不要な実行の回避
- 軽量なランナーイメージの選択

### 5. モニタリングツールは OSS を使う

アナリティクスにUmami（OSS）、SASTにSemgrep（OSS版）を使うことで、これらのツールコストをゼロまたは最小限に抑えています。

## 収益構造とbreakeven analysis

### 固定費 vs 変動費

| 種類 | 項目 | 月額 |
|------|------|------|
| 固定費 | Apple Developer Program | ¥1,250 |
| 固定費 | ドメイン | ¥500 |
| 固定費 | 共用サーバー | ¥100 |
| 固定費 | Umami ホスティング | ¥750 |
| 変動費 | Stripe手数料 | 取引額の3.6% |
| 変動費 | Gemini API | 従量課金 |
| 変動費 | Vercel Bandwidth超過 | 現時点では$0 |
| **固定費合計** | | **約¥2,600/月** |

月額の固定費が約2,600円ということは、**月にFocalizeの有料ユーザーが2〜3人いればインフラコストはカバーできる**計算です。MochiQの課金収入やShareTokuのアフィリエイト収入が加われば、さらに余裕が出ます。

### スケール時のコスト見通し

現在の無料枠ベースの構成は、ユーザー数が一定規模を超えると有料プランへの移行が必要になります。大まかな見通しは以下の通りです。

| トリガー | 対象 | 予想追加コスト |
|---------|------|-------------|
| Supabase 500MB超過 | Focalize or ShareToku | $25/月 |
| Vercel Bandwidth 100GB超過 | Focalize or ShareToku | $20/月 |
| Firebase Spark制限超過 | MochiQ or 元気ボタン | $25/月〜 |
| Gemini API 無料枠超過 | Focalize, MochiQ | 従量課金 |

これらのトリガーに到達するということは、ユーザー数が相当増えているということなので、それに伴う収益で十分にカバーできるはずです。「コストが増えるのは良いこと」。ユーザーが増えている証拠だからです。

## コスト管理の心構え

1人法人のコスト管理で最も重要なのは、**「コスト削減のために時間を使いすぎない」**ことです。

月額500円を節約するために半日かけてインフラを移行する。これは時間単価で考えれば明らかに損です。コスト削減は「仕組みとして無料枠に収まる設計」を最初にしておくことで実現し、日常的にコスト削減のための作業に時間を費やすことは避けるべきです。

## 「使えるアプリ」と「使い続けられるアプリ」は違う

高齢者向けのアプリを作ったことがある開発者なら、一度はこんな経験をしたことがあるのではないでしょうか。

「初回セットアップは手伝えたけど、1週間後には使われなくなっていた。」

機能としては動いている。バグもない。でも、親御さんのスマホでは起動すらされていない。

元気ボタンは、70代〜80代の方に「毎日ボタンを1回押してもらう」だけのアプリです。仕組みはシンプルですが、この「毎日使い続けてもらう」という要件は、想像以上に難しいものでした。

この章では、元気ボタンの開発を通して学んだ5つのUX原則を、Flutterの具体的な実装コードと共に解説します。

## 原則1: タップ領域は大きく — 44pxでは足りない

### AppleのHIGを超える基準が必要

Appleのヒューマンインターフェースガイドラインでは最低44pt、Material Designでは48dpを推奨しています。しかし、これらは健康な成人向けの最低ラインです。

加齢に伴い、指先の微細な制御能力は低下します。本態性振戦（手の震え）は65歳以上の約5%に見られ、タッチスクリーンに対する指の精度は若年者より有意に低いことが研究で示されています。

元気ボタンでは、メインのボタンサイズを**画面幅の65%**としています。

```dart
final screenWidth = MediaQuery.of(context).size.width;
final buttonSize = screenWidth * 0.65;
```

一般的なスマートフォンの画面幅を考えると、これは約240〜260pxに相当します。44pxの約6倍です。

「大きすぎないか」と思うかもしれません。しかし実際にテストしてみると、この大きさでも高齢者にとっては「押しやすい」と感じるレベルであり、「大きすぎる」という声は一度も出ませんでした。

### シニア向けタップ領域の目安

| 要素 | 一般的な推奨サイズ | シニア向け推奨サイズ |
|------|------------------|-------------------|
| メインアクション | 44〜48px | 64px以上（理想は80px以上） |
| 副次的ボタン | 36〜44px | 48〜64px |
| テキストリンク | 44px（タップ領域） | 使用を避ける |
| ナビゲーション | 44px | 56px以上 |

特に重要なのが**ボタン同士の間隔**です。タップ領域を大きくしても、ボタンが密集していればミスタップは防げません。要素間の余白は最低16px、理想的には24px以上確保してください。

テキストリンク（下線付きのテキスト）はシニア向けUIではなるべく使わないことを推奨します。リンクと通常テキストの視覚的な区別がつきにくく、タップ領域も小さくなりがちです。代わりにボタンコンポーネントを使いましょう。

## 原則2: 情報量を最小限に — 「1画面1アクション」

### ヒックの法則とシニアの離脱

選択肢の数が増えると、意思決定にかかる時間は対数的に増加します（ヒックの法則）。高齢者の場合、この影響がより顕著に表れます。さらに「時間がかかる」だけでなく「不安になる」という心理的影響があります。「どれを押せばいいかわからない」→「間違えたら壊れるかもしれない」→「もう触らない」。

元気ボタンの親側画面は、以下の要素だけで構成されています。

- 時刻表示（大きなフォント）
- 日付表示
- 1つのボタン（「今日も元気！」）
- 1週間のチェックイン状況（ドット表示）

メニューバーもハンバーガーメニューもありません。設定画面への導線は子ども側のアプリに集約し、親側の画面はアプリを開いた瞬間に「やるべきことが1つだけ」とわかる設計です。

### フォントサイズの設計

```dart
// 親側画面のフォントサイズ設定
static TextStyle get parentTime => _baseStyle.copyWith(
      fontSize: 64,  // 時刻表示: 64px
      fontWeight: FontWeight.w700,
    );

static TextStyle get parentButton => _baseStyle.copyWith(
      fontSize: 28,  // ボタンテキスト: 28px
      fontWeight: FontWeight.w700,
    );

static TextStyle get parentHelper => _baseStyle.copyWith(
      fontSize: 18,  // 補助テキスト: 18px
      fontWeight: FontWeight.w400,
    );
```

一般的なアプリのボディテキストは14〜16px程度ですが、親側画面では最小でも18pxとしています。

### 「あると便利」を削る判断基準

開発を進めると、「履歴を見られるようにしよう」「天気予報も表示しよう」「家族へのメッセージ機能を追加しよう」とアイデアが湧いてきます。それぞれは合理的に見えますが、要素を1つ追加するたびに認知的複雑さは増します。

元気ボタンでは、機能追加の判断基準を1つだけ設けています。

**「その機能がないと、明日もボタンを押してもらえないか？」**

この問いにNoと答えられる機能は、すべて削っています。

## 原則3: 色のコントラスト比を高く

### 加齢に伴う色覚の変化

人間の水晶体は加齢とともに黄色く変色していきます。60歳を過ぎるとこの変色は顕著になり、青色や紫色の識別が特に困難になります。白内障は80歳以上のほぼ全員に何らかの程度で発症し、コントラスト感度の低下を引き起こします。

### WCAG AAA基準（7:1以上）を標準に

WCAGのコントラスト比基準:
- **AA（通常テキスト）:** 4.5:1以上
- **AA（大きなテキスト）:** 3:1以上
- **AAA（通常テキスト）:** 7:1以上

シニア向けアプリでは**AAA基準を標準として設計する**ことを推奨します。

```dart
// 元気ボタンのカラーパレット
static const textPrimary = Color(0xFF2D2D2D);   // メインテキスト
static const textSecondary = Color(0xFF6B6B6B);  // 補助テキスト
static const background = Color(0xFFFFF9F0);     // クリーム系の柔らかい白
static const primary = Color(0xFFFF8C42);         // 温かいオレンジ
```

背景色（`#FFF9F0`）とテキスト色（`#2D2D2D`）のコントラスト比は約13:1で、AAA基準を大幅に超えています。

### 色だけに頼らない情報伝達

高齢者の中には色覚に変化がある方も多いため、色だけで状態を伝えることは避けるべきです。

```dart
if (widget.isCheckedIn) ...[
  const Icon(
    Icons.check_circle,
    size: 48,
    color: AppColors.checkedText,
  ),
  const SizedBox(height: 8),
  Text(
    '届けました！',
    style: AppTypography.parentButton.copyWith(
      color: AppColors.checkedText,
    ),
  ),
]
```

「オレンジのボタンが緑に変わった」だけでは伝わらない方がいます。チェックマークアイコンとテキストを併用することで、色が正確に見えなくても状態が明確に理解できます。

### デザインチェックの習慣

1. **コントラスト比の計測** — WebAIMのContrast Checkerで数値を確認する。感覚に頼らない
2. **グレースケールテスト** — 画面全体をグレースケールに変換しても、情報が伝わるか確認する
3. **ぼかしテスト** — 画面を軽くぼかした状態でも、主要な要素の区別がつくか確認する（白内障のシミュレーション）

## 原則4: フィードバックを明確に — 3チャネルで「押せた」を伝える

### シニアの最大の不安は「操作が成功したかわからない」こと

若い世代であれば、0.3秒の微細なアニメーションで「成功した」と判断できます。しかし、スマートフォンの操作経験が浅い高齢者は、その微細な変化を見逃しやすい。

「押したのに何も起きていない気がする」→「もう一度押してみよう」→「何か変なことになったかもしれない」→「怖いからもう触らない」

この心理的な流れが、アプリが使われなくなる最大の原因です。

### マルチモーダルなフィードバック

元気ボタンでは、ボタン押下時に**3つのチャネル**でフィードバックを返しています。

**視覚フィードバック:** ボタンの外観が明確に変化する。色がオレンジから緑へ、テキストが「今日も元気！」から「届けました！」に変わり、チェックマークアイコンが表示される。

**触覚フィードバック:** 端末がバイブレーションで応答する。

```dart
final hasVibrator = await Vibration.hasVibrator();
if (hasVibrator == true) {
  Vibration.vibrate(duration: 50);
}
```

**アニメーション:** ボタンが押し込まれるスケールアニメーションで、物理的な「押した感覚」を演出する。

```dart
_scaleAnimation = Tween<double>(
  begin: 1.0,
  end: 0.95,
).animate(CurvedAnimation(
  parent: _animationController,
  curve: Curves.easeInOut,
));
```

さらに、成功を伝えるダイアログも表示します。通信中はボタン上にCircularProgressIndicatorを表示し、タップ操作も無効化して「今、処理をしています」ということを視覚的に明示します。

### ミスタップの許容

フィードバックと合わせて重要なのが、「取り返しのつかない操作」を作らないことです。元気ボタンは1日1回の安否確認なので、誤って押してしまっても、その日の安否確認が完了するだけです。この「ミスタップしても問題ない」設計が、操作への心理的ハードルを大きく下げています。

## 原則5: 認知負荷を下げる — 「考えさせない」デザイン

### 曖昧な表現を排除する

| 避けるべき表現 | 代わりに使う表現 |
|--------------|---------------|
| 送信 | 元気を届ける |
| キャンセル | やめる |
| 設定 | 変更する |
| ログアウト | 終了する |
| 同期中 | 読み込んでいます |
| ステータス | 状態 |

元気ボタンのボタンテキストは「送信」ではなく「今日も元気！」です。押した後は「送信完了」ではなく「届けました！」と表示します。技術的な用語を一切使わず、日常会話の延長線上にある言葉を選んでいます。

### 一貫性の徹底

画面Aでは「右上のボタンで進む」のに、画面Bでは「下のボタンで進む」。こういった不一致は高齢者にとって混乱の原因になります。元気ボタンの初期設定フローでは以下を徹底しました。

- 「次へ進む」ボタンは常に画面下部中央
- すべてのボタンは同じスタイル（角丸・同じ色・同じサイズ）
- 戻る操作は常に左上

### 時間的制約を設けない

元気ボタンのボタンは1日中いつでも押せます。「朝8時〜9時の間に」といった制約は設けていません。時間に追われる感覚を与えないことで、自分のペースで操作できる安心感を提供しています。

### 慣れ親しんだメンタルモデルを活用する

元気ボタンの大きな丸いボタンは、「非常ベルのボタン」や「エレベーターのボタン」と同じメンタルモデルです。「大きな丸いものは押せる」という直感的な理解は世代を問わず共通しています。

フラットデザインは見た目の洗練さでは優れていますが、「押せるかどうかが見た目でわかる」という点では従来の立体的なボタンのほうがシニアに親切です。元気ボタンでは、box-shadowを適用して浮き上がっている感覚を演出しています。

```dart
boxShadow: [
  BoxShadow(
    color: AppColors.primary.withValues(alpha: 0.3),
    blurRadius: 20,
    offset: const Offset(0, 10),
  ),
],
```

## シニアUXの原則は全プロダクトに応用できる

5つの原則を振り返ります。

1. **タップ領域は大きく** — 44pxではなく64px以上。間隔も十分に確保
2. **情報量を最小限に** — 1画面1アクション。「あると便利」は削る
3. **色のコントラスト比を高く** — WCAG AAA基準（7:1以上）を標準に
4. **フィードバックを明確に** — 視覚・触覚・アニメーションの3チャネルで伝える
5. **認知負荷を下げる** — 技術用語を排除し、一貫性を保ち、時間制限を設けない

これらに共通するのは「引き算のデザイン」です。機能を追加するのではなく削る。選択肢を増やすのではなく絞る。

この考え方はシニア向けアプリに限った話ではありません。FocalizeのダッシュボードUIでも、元気ボタンで鍛えた「認知負荷を最小限にする」思考が活きています。情報過多でユーザーの注意力が年々短くなっている現代において、引き算の設計はあらゆるプロダクトに求められています。

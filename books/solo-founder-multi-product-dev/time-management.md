## 1人開発最大のボトルネックは「時間」

技術的な問題はAIの力を借りて解決できます。コストは無料枠の活用で最小化できます。しかし、1日24時間という制約だけは誰にも変えられません。

5プロダクトを1人で回す上で最大のボトルネックは、常に時間です。5つ全てに毎日均等に時間を割くのは不可能であり、仮にそうしても各プロダクトに1日1時間程度しか使えず、どれも中途半端になります。

この章では、限られた時間を最大限に活用するための実践的な手法を共有します。

## 週次フォーカスバッチング

### コンテキストスイッチのコストを最小化する

5プロダクトを毎日均等に触るのは非効率です。プロダクトAからBに切り替えるとき、コードベースの理解、現在のタスク状況の把握、開発環境の切り替えなど、最低30分のコンテキストスイッチコストが発生します。

1日に3回プロダクトを切り替えたら、90分が切り替えだけで消えます。これを週5日続けると、週7.5時間。作業可能な時間が仮に週40時間だとすると、約20%がコンテキストスイッチに消費されることになります。

### 週単位でフォーカスするプロダクトを決める

代わりに採用しているのが、**週単位でフォーカスするプロダクトを決める**方法です。

```
月間スケジュールの例:

Week 1: Focalize (主要SaaS。機能追加・バグ修正)
Week 2: MochiQ (アプリアップデート + LP改善)
Week 3: ShareToku + 元気ボタン (メンテナンス + 小規模改善)
Week 4: Skill Tracker + 全体横断 (LP改善 + インフラ更新)
```

### フォーカスの濃淡

全てのプロダクトに同じ時間を割く必要はありません。プロダクトのフェーズに応じて、フォーカスの濃淡をつけています。

| プロダクト | フェーズ | 月間の時間配分目安 |
|-----------|---------|-----------------|
| Focalize | 収益化 | 40% |
| MochiQ | 成長期 | 25% |
| 元気ボタン | 成長期 | 15% |
| ShareToku | MVP | 10% |
| Skill Tracker | 企画段階 | 10% |

Focalizeが最も多くの時間を占めるのは、現在収益化フェーズにあり、有料ユーザーへの責任があるからです。バグ修正や機能リクエストへの対応は、他のプロダクトより優先度が高くなります。

### 「割り込み」のルール

バグ報告やエスカレーションがあればスケジュールは崩れます。その場合でも以下のルールを設けています。

1. **緊急度の判定**: ユーザーが使えない障害 → 即対応。表示の軽微な不具合 → 次の該当週に対応
2. **タイムボックス**: 割り込み対応は2時間以内に収める。2時間で解決しない場合はworkaroundを提示し、根本対応は次の該当週に回す
3. **記録**: 割り込みの内容と対応時間を記録し、月末に振り返る

## 1日のタイムブロック

### 集中作業とルーティンを分離する

1日の時間配分も意識的に設計しています。

```
1日のタイムブロック:

06:00 - 07:00  朝のルーティン
07:00 - 07:30  メール・通知チェック、タスクの優先順位確認
07:30 - 12:00  深い集中作業（コーディング、設計）
12:00 - 13:00  昼休憩
13:00 - 15:00  中程度の集中作業（コードレビュー、テスト、ドキュメント）
15:00 - 16:00  軽作業（メール返信、SNS、管理タスク）
16:00 - 17:00  翌日の計画、振り返り
```

最も重要なのは、**午前中の4.5時間を深い集中作業に確保する**ことです。この時間帯にSlackやメールを見ない。通知を切る。新機能の実装や複雑なバグの修正など、高い認知負荷を要するタスクはすべてこの時間帯に行います。

### 「15分ルール」

タスクに着手する前に「15分で終わるか」を判断します。15分以内で終わるタスク（typoの修正、依存パッケージのアップデート、コメントの追加など）はその場で片付けます。15分以上かかるタスクは、該当プロダクトの週に回します。

この判断基準により、「ちょっとした修正」がコンテキストスイッチの引き金になることを防いでいます。

## AI活用の実態

### Claude Code — 事実上のチームメンバー

AIは「便利なツール」ではなく、**事実上のチームメンバー**です。1人開発の生産性を2〜3倍に引き上げている最大の要因がAI活用です。

**コーディング:** Claude Codeを開発の中核に据え、機能実装からテスト作成、リファクタリングまで一貫して活用しています。特に、新規機能の実装では「仕様を自然言語で記述 → AIがコード生成 → 自分がレビュー・修正」というフローが定着しています。

```
AI活用のレイヤー:

┌──────────────────────────────────────┐
│  経営判断: AI-CEO Framework          │
│  (プロダクト横断の優先順位管理)        │
├──────────────────────────────────────┤
│  開発: Claude Code                   │
│  (実装、テスト、リファクタリング)      │
├──────────────────────────────────────┤
│  品質保証: Claude Code Action        │
│  (PR自動レビュー)                    │
├──────────────────────────────────────┤
│  プロダクト機能: Gemini AI            │
│  (Focalize: メール解析, MochiQ: クイズ生成) │
└──────────────────────────────────────┘
```

**コードレビュー:** Claude Code ActionによるPR自動レビューに加え、ローカルでもClaude Codeに既存コードの品質チェックを依頼しています。1人開発では「自分のコードを自分でレビューする」バイアスが避けられません。AIはこのバイアスを軽減するセーフティネットです。

### AI-CEO Framework — プロダクト横断の意思決定支援

AI-CEO Frameworkという自作の経営支援フレームワークを構築し、5プロダクトの状態管理・優先順位判断をAIエージェント群に委任しています。

6部門（開発、マーケ、営業、財務、CS、法務）に対応するAIサブエージェントがファイルベースで情報を管理し、以下を自動化しています。

- **朝のダイジェスト生成**: 各プロダクトの状態、承認待ちタスク、KPIサマリを朝一に確認
- **優先順位の提案**: 「今週はFocalizeのバグ修正を優先すべき」といった提案
- **意思決定ログ**: 経営判断の履歴をファイルとして蓄積し、後から振り返り可能に

これにより、「今日何をすべきか」の判断にかかる時間を大幅に短縮できています。

### AIに委任できること、できないこと

AI活用で注意しているのは、**AIに委任する範囲を明確にする**ことです。

| AIに委任できる | AIに委任できない |
|-------------|---------------|
| コード生成、テスト作成 | プロダクトのビジョン決定 |
| コードレビュー | ユーザーの声の解釈 |
| 定型的なドキュメント作成 | 撤退/ピボットの判断 |
| バグの原因調査 | 法的リスクの最終判断 |
| リファクタリング提案 | パートナーとの交渉 |

AIはコードを書けますが、「このプロダクトを続けるべきか」「このユーザーの要望にどう応えるか」という判断は自分で行う必要があります。AIは実行を加速させるツールであり、意思決定を代替するものではありません。

## 意図的な技術的負債の管理

### 「完璧」を目指さない

5プロダクトを1人で運用する場合、すべてのプロダクトを完璧な状態に保つことは不可能です。意図的に技術的負債を受け入れ、それを管理する仕組みが必要です。

### 技術的負債の3カテゴリ

| カテゴリ | 定義 | 対応方針 |
|---------|------|---------|
| 即時対応 | セキュリティ脆弱性、データ損失リスク | 発見次第対応。他の作業を中断してでも修正 |
| 計画的返済 | テストカバレッジ不足、古いバージョンの依存関係 | 四半期ごとに「負債返済週」を設けて計画的に対応 |
| 許容する負債 | コードの重複、理想的でないアーキテクチャ | ユーザーに影響がない限り、当面は許容 |

重要なのは**許容する負債を明示的に記録すること**です。「ここは意図的に負債として残している」というコメントをコードに残すことで、将来の自分が「これはバグなのか、意図的な判断なのか」で悩む時間を節約できます。

```typescript
// TODO(tech-debt): MochiQのProvider → Riverpod移行
// 元気ボタンとの統一のため移行予定だが、現在動作に問題ないため優先度低
// 次のメジャーアップデート時に対応する
```

### 四半期の「負債返済週」

3か月に1回、通常の機能開発を止めて「負債返済週」を設けています。この週は新機能を一切作らず、以下に集中します。

- 依存パッケージのアップデート
- テストカバレッジの改善
- リファクタリング
- ドキュメントの更新
- CI/CDパイプラインの改善

## 休息の設計

### 1人開発の最大リスクは「自分が倒れること」

5プロダクトの運用に必死になると、休息を後回しにしがちです。しかし、1人法人では自分が倒れたら全てが止まります。これは技術的リスクではなく、事業継続のリスクです。

### 具体的なルール

- **週に1日は完全オフ**: プロダクトのことを一切考えない日を設ける
- **年に4回は1週間以上の休暇**: 事前に顧客に周知し、緊急連絡先のみ残す
- **夜10時以降はコードを書かない**: 夜中に書いたコードは翌朝の自分が苦しむ

これらは「理想」ではなく、守らないと数か月後にバーンアウトすることを経験的に学んだ上での「必須ルール」です。

## タイムマネジメントのまとめ

1. **週次フォーカスバッチング**: プロダクトを毎日切り替えず、週単位でフォーカスする
2. **午前中の集中時間を死守**: 通知を切り、深い作業に4.5時間を確保する
3. **AIをチームメンバーとして活用**: コーディング、レビュー、経営判断の全レイヤーで
4. **技術的負債を3カテゴリで管理**: 即時対応、計画的返済、許容する負債
5. **休息を設計する**: 自分が最大のSPOF（単一障害点）であることを自覚する
